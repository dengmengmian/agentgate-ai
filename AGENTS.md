# 项目定位
Otaro = 多模型兼容的本地 coding agent workbench

# Agent 项目规则

本文件是本项目的 AI 编程 Agent 执行规范。

适用于：
- Claude Code：`CLAUDE.md`
- Codex / OpenAI Codex：`AGENTS.md`
- OpenCode / 其他支持 Markdown 项目指令的 Agent

每次新任务开始前，必须先读取并遵守本文件。

---

## 1. 基本原则

1. 先判断真实需求，再判断实现方式。
2. 如果需求动机不合理、实现路径明显不优、会破坏长期可维护性，必须先停下并说明原因。
3. 优先选择能解决真实问题的最小正确改动。
4. 不允许为了完成当前任务破坏正确性、一致性和可维护性。
5. 不允许用兜底逻辑、假数据、静默降级掩盖真实错误。
6. 输出必须结论先行，直接、简洁、有证据。
7. 本项目默认没有历史包袱。用户没有明确要求兼容旧行为、旧协议、旧字段或旧 UI 时，必须按新的正确方案一次性改到位，不允许额外保留兼容层、双轨逻辑、旧入口或旧数据结构。
8. 如果新方案与旧实现冲突，优先迁移到新方案；只有在用户明确要求兼容、迁移成本过高且已说明风险时，才允许临时保留兼容代码，并必须标注清理计划。

---

## 2. 任务分级

### 2.1 小任务

同时满足以下条件，才算小任务：

- 只影响 1～3 个文件；
- 不改变架构；
- 不改变接口协议；
- 不改变数据库结构；
- 不改变权限或认证逻辑；
- 不引入新依赖；
- 不影响公共组件、公共工具、公共抽象。

小任务可以在读取相关文件后直接修改。

执行流程：

1. 读取相关文件；
2. 做最小正确改动；
3. 运行必要验证；
4. 检查 diff；
5. 汇报完成情况、验证方式、风险和使用的工具。

---

### 2.2 大任务

满足任一条件，就算大任务：

- 影响超过 3 个文件；
- 涉及超过 1 个模块；
- 涉及重构、迁移、统一规范、一致性清理；
- 涉及公用数据库；
- 涉及公用接口返回结构；
- 涉及公共组件、公共 hooks、公共工具、公共服务；
- 需要对比或统一多个项目、多个模块；
- 可能影响已有业务行为。

大任务必须先给计划，再执行。

计划必须包含：

| 项目 | 要求 |
|---|---|
| 目标 | 本次任务解决什么问题 |
| 文件 | 需要读取和可能修改的关键文件 |
| 动作 | 每个区域准备做什么 |
| 验证 | 用哪些命令或检查证明没有改坏 |
| 完成标准 | 什么结果代表任务完成 |
| 风险 | 可能影响什么，还有什么不确定 |

计划不清楚时，不允许直接实现大任务。

---

### 2.3 架构级任务

满足任一条件，就算架构级任务：

- 改模块边界；
- 改接口协议或返回结构；
- 改数据库结构或迁移策略；
- 改认证、授权、权限模型；
- 改核心运行时、框架、基础设施或部署模型；
- 引入新的基础抽象；
- 大范围替换现有实现；
- 需要做长期兼容性决策。

架构级任务必须走 OpenSpec。

OpenSpec 流程：

1. 写 proposal；
2. 等用户明确确认；
3. 确认后才执行；
4. 完成后归档。

禁止跳过用户确认直接实现架构级变更。

---

## 3. 修改代码前要求

修改代码前，必须明确：

| 项目 | 要求 |
|---|---|
| 目标 | 当前到底要解决什么问题 |
| 文件 | 需要读取或修改哪些文件 |
| 动作 | 准备改什么 |
| 验证 | 如何验证结果 |
| 完成标准 | 什么算完成 |

规则：

1. 禁止在未读取相关真实文件前修改代码。
2. 禁止只凭记忆、猜测、文件名或历史对话修改代码。
3. 如果业务含义不明确，必须说明不确定点，并选择最小安全改法。
4. 如果用户当前要求与本文件冲突，必须先停下并说明冲突点。

---

## 4. 严格禁止

除非用户明确确认，否则禁止以下行为：

1. 在生产代码中使用 Mock 数据。
2. 使用假数据模拟真实行为。
3. 硬编码结果，让功能看起来完成。
4. 添加掩盖真实错误的兜底逻辑。
5. 静默吞掉错误。
6. 绕过真实业务逻辑让测试通过。
7. 未评估影响就改变公共 API 行为。
8. 未批准方案就改变数据库结构。
9. 未确认就创建新分支。
10. 未确认就创建新 worktree。
11. 未确认就删除文件。
12. 未确认就执行破坏性 git 命令。
13. 未确认就写数据库。
14. 未确认就安装新依赖。
15. 未确认就修改部署、代理、证书或生产配置。
16. 只修一个表面问题，却不检查同类问题。
17. 未验证就汇报完成。
18. 没有证据时说“应该可以”“大概修好了”“看起来没问题”。

## 5. 命令和权限

### 5.1 可以直接执行的命令

以下只读或低风险命令可以直接执行：

```bash
pwd
ls
find
cat
sed -n
grep
rg
git status
git diff
git log
git show
curl
```

以下验证命令在合适时可以直接执行：

```bash
npm test
npm run test
npm run build
pnpm test
pnpm build
pnpm lint
go test
go test ./...
go build
cargo test
cargo build
```

### 5.2 必须先确认的操作

以下操作必须先获得用户明确确认：

| 操作 | 原因 |
|---|---|
| `rm` / `rm -rf` | 破坏性操作 |
| `git reset` | 可能丢失修改 |
| `git checkout .` | 可能覆盖用户改动 |
| `git clean` | 可能删除文件 |
| 创建分支 | 改变工作流 |
| 创建 worktree | 改变工作区结构 |
| 写数据库 | 可能污染真实数据 |
| 数据库迁移 | 可能影响数据完整性 |
| 安装依赖 | 改变依赖图 |
| 修改部署配置 | 可能影响生产 |
| 修改代理 / Nginx / 证书配置 | 可能影响访问 |
| 修改 CI/CD 配置 | 可能影响发布流程 |

## 6. 子代理 / 并行分析规则

工具支持时，复杂任务应该拆给子代理或并行分析。

满足任一条件，就算复杂任务：

- 涉及 5 个以上文件；
- 涉及 2 个以上模块；
- 需要对比多个项目或模块；
- 需要分析架构、权限、数据库、接口；
- 需要做大范围一致性审查；
- 需要排查难以复现的问题；
- 需要阅读大量代码。

子代理规则：

1. 子代理只负责调研、对比、定位问题。
2. 子代理不做最终架构决策。
3. 主 Agent 负责汇总结论、取舍和执行计划。
4. 主上下文只保留结论、决策和可执行步骤。

## 7. lessons.md 规则

如果用户指出错误、遗漏、误判或返工点，必须更新 `lessons.md`。

如果 `lessons.md` 不存在，需要创建。

每条 lesson 必须包含：

| 项目 | 内容 |
|---|---|
| 问题 | 这次错在哪里 |
| 原因 | 为什么会错 |
| 规则 | 下次应该怎么避免 |
| 适用范围 | 哪类任务需要注意 |

每个新任务开始前：

1. 如果 `lessons.md` 存在，必须先读取；
2. 把相关经验纳入当前任务；
3. 不允许重复已经记录过的错误。

---

## 8. 排查顺序

网络、证书、代理、超时、请求失败、API 访问问题，必须按以下顺序排查：

1. 请求入口；
2. 客户端请求 URL 和方法；
3. 代理 / Nginx / 网关；
4. TLS / 证书；
5. 域名 / DNS；
6. 环境变量；
7. 服务端路由注册；
8. 中间件；
9. 业务代码；
10. 第三方 API。

禁止一开始就改业务代码。

---

## 9. 代码修改规则

修改代码时必须遵守：

1. 优先复用项目现有模式。
2. 非必要不引入新模式。
3. 命名、目录结构、错误处理、返回结构必须一致。
4. 公共逻辑必须放在公共层。
5. 禁止把公共逻辑复制粘贴到多个地方。
6. 改动范围要尽量小，但不能留下同类问题。
7. 除非用户明确要求兼容旧行为，否则不要为了“兼容”保留旧实现；涉及协议、运行时、工具结果、上下文治理、模型适配、UI 交互等核心路径时，应统一切到新路径。
8. 修改后必须检查 `git diff`。
9. 必须用最相关的测试、构建、lint 或人工检查验证。
10. 如果无法验证，必须说明原因和剩余风险。

---

## 10. API 和数据库规则

处理 API 时：

1. 请求和返回结构必须在模块间保持一致。
2. 如果项目已有统一返回格式，禁止新增另一套返回格式。
3. 禁止混淆传输层错误和业务错误。
4. 改公共 API 行为前，必须说明影响范围。
5. 禁止静默忽略上游错误。
6. 新协议落地时必须以新协议为唯一主路径；除非用户明确要求，禁止长期同时维护新旧两套 API、工具协议或 provider 适配路径。

处理数据库时：

1. 未确认禁止写真实数据库。
2. 未批准方案禁止改数据库结构。
3. 未批准 schema 变更，禁止创建 migration 文件。
4. 禁止用兜底查询掩盖数据一致性问题。
5. 改数据访问代码前，必须检查现有 repository、transaction、错误处理模式。

---

## 11. 前端和 UI 规则

处理前端代码时：

1. 优先复用现有组件、布局模式、设计 token 和样式规范。
2. 如果同类组件都有同样问题，禁止只修一个组件。
3. 间距、字体、圆角、阴影、颜色、交互状态必须统一。
4. 未确认禁止引入新的 UI 库。
5. 禁止把业务数据硬编码进 UI 组件。
6. Mock 数据只允许用于独立原型、演示或用户明确要求的 mock 预览。
7. 生产 UI 必须连接真实数据路径，除非任务明确要求不接。

---

## 12. OpenSpec 规则

架构级变更必须使用 OpenSpec。

流程：

1. Proposal；
2. 用户确认；
3. 实现；
4. 验证；
5. 归档。

Proposal 必须包含：

| 项目 | 要求 |
|---|---|
| 问题 | 当前哪里不合理 |
| 目标 | 这次变更要达到什么效果 |
| 范围 | 改什么，不改什么 |
| 设计 | 技术方案 |
| 影响 | API、数据库、UI、部署、兼容性影响 |
| 验证 | 如何证明成功 |
| 回滚 | 如果做错，怎么恢复 |

用户未确认 proposal 前，禁止实现。

---

## 13. 输出格式

所有回复必须结论先行。

多项内容优先使用 Markdown 表格。

完成任务后必须输出：

| 项目 | 内容 |
|---|---|
| 完成情况 | 做了什么 |
| 修改文件 | 改了哪些文件 |
| 验证方式 | 执行了哪些命令或检查，结果如何 |
| 风险 | 还有什么不确定或未覆盖 |
| 使用工具 | 使用了哪些工具；没有则写“无” |

禁止：

1. 复读用户已经知道的背景。
2. 空泛汇报。
3. 大段解释自己多努力。
4. 没证据就保证结果。
5. 未验证却说任务完成。

---

## 14. 冲突处理

如果用户指令和本文件冲突：

1. 先停止冲突操作；
2. 说明冲突点；
3. 请求用户明确确认；
4. 用户确认后才能继续。

如果工具、插件、skill、子代理指令和本文件冲突，以本文件为项目规则，除非用户明确说以其他规则为准。

---

## 15. 最终自检

每次任务结束前，必须检查：

| 检查项 | 要求 |
|---|---|
| 是否读取相关文件 | 是 |
| 是否符合真实需求 | 是 |
| 是否没有新增 Mock / 假数据 / 兜底逻辑 | 是 |
| 是否遵守现有项目模式 | 是 |
| 是否检查 diff | 是 |
| 是否验证或说明无法验证原因 | 是 |
| 是否说明风险 | 是 |
| 用户指出错误时是否更新 lessons.md | 是 |
