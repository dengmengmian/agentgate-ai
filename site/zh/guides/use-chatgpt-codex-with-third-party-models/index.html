1.4.12 新能力 · ChatGPT+Codex

合并版 ChatGPT+Codex，照样跑 DeepSeek / MiMo，工具还能真调。

OpenAI 把 ChatGPT 和 Codex 合成一个桌面端。gpt-5.6 协议把工具定义藏进 additional_tools。不在本地做提升，第三方模型就收不到工具 schema，只能编假 <tool_call>。AgentGate 1.4.12 在你自己的机器上把这件事修好——不是云上黑盒。

一句话

合并版客户端只配一次本地入口；插件需要时保留官方登录；DeepSeek / MiMo / Kimi / GLM / OpenRouter 等上游在 AgentGate 里切换，工具调用走真实 schema。

合并之后哪里断了

新桌面端仍然走大家熟悉的 OpenAI Responses 路径，但工具定义不再总是顶层 tools。它们会出现在 input 数组里，类型是 additional_tools。 

合并版 ChatGPT+Codex (gpt-5.6)
  -> Responses 请求
  -> 工具在 input[].type = "additional_tools"   ← 很容易被忽略
  -> 朴素代理只转发空 tools
  -> DeepSeek / MiMo / … 用假 <tool_call> 文本应付

这不是模型变笨，是模型根本没拿到工具 schema。升级客户端后“工具全坏了”，密钥往往没问题，坏在协议形态。

AgentGate 1.4.12 做了什么

问题 本地网关怎么修
工具只在 additional_tools 转换前提升合并进 tools，覆盖 Chat / Anthropic / Gemini 上游
gpt-5.6 模型名对不上 目录与价格补齐 gpt-5.6 / -sol / -terra / -luna，Codex 推荐映射同步
假 <commentary> / <context_addition> 标签 流式跨 chunk 剥离标签，保留可读正文
合并版进程名变了 客户端重启工具识别主进程 “ChatGPT”，并兼容旧 Codex.app

本地链路

ChatGPT+Codex 桌面端
  -> OpenAI 风格 Provider 入口（需要时保留插件 / 账号路径）
  -> http://127.0.0.1:9090  (AgentGate)
  -> DeepSeek / 小米 MiMo / OpenAI / Kimi / GLM / DashScope / OpenRouter / …

AgentGate 不是托管转售。Key 在你机器上。日志能看到路由、转换后的请求、token、费用和 failover——挂了能看见挂在哪，不是黑盒。

五分钟上手

  1. 安装 AgentGate 1.4.12 （macOS / Windows / Linux）。
  2. 添加 Provider（DeepSeek、MiMo …），粘贴 API key。
  3. 启动网关，默认 http://127.0.0.1:9090。
  4. 在 客户端 对 Codex / ChatGPT+Codex 点 应用配置。
  5. 发一条会触发工具的消息，在 日志 里确认上游和 tools 存在——不是只有纯文本。

还要插件和官方账号能力？保持桌面端登录；AgentGate 保留 OpenAI 鉴权那条 Provider 路径，只把模型 base URL 指到本机。细节见 Codex Desktop + 插件。

适合谁

  • 升了合并版客户端，第三方模型突然“不会用工具”。
  • 希望 Codex / Claude Code / Gemini CLI / OpenCode 共用一个本地入口，而不是每个 App 改一套代理。
  • 需要 failover、费用追踪、一键切回官方配置。

常见问题

合并后为什么工具突然坏了？

工具定义挪进了 input 里的 additional_tools。只读顶层 tools 的代理会把空 schema 发给上游。

这是云端代理吗？

不是，跑在本机，自带 key。要多租户计费的服务端方案，看 LiteLLM / new-api 一类——对比见 AgentGate vs LiteLLM。

要哪个版本？

1.4.12 及以上 才有 additional_tools 提升和 gpt-5.6 目录。更早版本仍可当通用网关，但修不了这种协议形态。
